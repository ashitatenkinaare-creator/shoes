# Sneaker Radar — 製品要件（確定版）

ユーザ変更要望に対するヒアリング結果（2026-07-05）を反映したスコープ定義。
現行実装（KicksDB 同期・ブランド/レア/コラボフィルタ・ウォッチリスト・アプリ内通知）を前提とする。

---

## 1. 背景・課題

SNKRDUNK 等のタイムライン型アプリでは、欲しい情報が流れて見逃しやすい。
**条件に合う商品を一覧で固定表示**し、**抽選・発売を逃さない**ことが目的。

---

## 2. スコープ外（今回対象外）

| 項目                                     | 理由                                        |
| ---------------------------------------- | ------------------------------------------- |
| SNS 監視（X / Instagram 等）             | 工数・API コスト・運用負荷が大きい          |
| スニーカー以外（Gパン、Gジャン、時計等） | データソース・スキーマ・UI の全面拡張が必要 |
| Google カレンダーへの完全自動追加        | ユーザ確認なしの書き込みは不要（後述）      |

---

## 3. スコープ内（MVP 以降）

### 3.1 監視対象・フィルタ

| 要件              | 内容                               | 技術方針                                                       |
| ----------------- | ---------------------------------- | -------------------------------------------------------------- |
| メーカー          | Nike、Converse 等                  | **既存** `user_preferences.brands`                             |
| モデル/シルエット | All Star、One Star 等              | **新規** `silhouettes[]` またはモデル名キーワードフィルタ      |
| コラボブランド    | KITH、Brooks Brothers、BEAMS       | **拡張** `config/rare-collab-keywords.json` + 明示的コラボタグ |
| 情報ソース（HP）  | **各ブランド・ストアの抽選ページ** | **新規** ソース登録 + 定期取得パイプライン（後述）             |

**HP の定義:** ブランド公式トップではなく **抽選・エントリー・発売告知ページ** を指す。

### 3.2 表示 UX（タイムライン問題の回避）

| 要件           | 内容                     | 技術方針                                        |
| -------------- | ------------------------ | ----------------------------------------------- |
| 条件マッチ一覧 | 自分の設定に合う商品のみ | **既存** ダッシュボード + フィルタ強化          |
| 流れない UI    | ウォッチリスト・条件保存 | **既存** + ピン留め・ダイジェスト通知の拡張余地 |

### 3.3 類似ユーザーの注目（任意 ON/OFF）

| 要件         | 内容                                         | 技術方針                                             |
| ------------ | -------------------------------------------- | ---------------------------------------------------- |
| データソース | **自アプリ内の行動**（ウォッチ、詳細閲覧等） | SNS 外部データは使わない                             |
| 設定         | 表示する / しない                            | **新規** `user_preferences.show_similar_users`（仮） |
| ロジック     | 同ブランド・同コラボ傾向の集計               | 匿名集計テーブル + ダッシュボード混在表示            |

※ ユーザー数が少ない初期は「同じ商品を N 人がウォッチ」程度の簡易版から開始。

### 3.4 Google カレンダー

| 要件         | 内容                                     | 技術方針                                                |
| ------------ | ---------------------------------------- | ------------------------------------------------------- |
| 追加方式     | **確認してから追加**（完全自動はしない） | 通知 or 詳細画面で「カレンダーに追加しますか？」        |
| 連携         | Google Calendar API + OAuth 2.0          | ユーザーが Google 連携後、確認ダイアログ → イベント作成 |
| イベント内容 | 発売日・抽選締切（取得できれば）         | `radar_sneakers` + 抽選ソースの日付フィールド           |

## 4. データソース構成（確定）

```
┌─────────────────────────────────────────────────────────┐
│                    Sneaker Radar                         │
├─────────────────────────────────────────────────────────┤
│  ソース A: KicksDB / StockX（既存）                      │
│    - 新作カタログ、画像、概算発売日                       │
│    - product_type = sneakers のみ                        │
├─────────────────────────────────────────────────────────┤
│  ソース B: 抽選ページ（新規・Phase 2）                   │
│    - Nike SNKRS、Converse 抽選、BEAMS/KITH 等の登録 URL │
│    - 定期クロール / RSS（あれば）→ radar テーブルへ upsert│
├─────────────────────────────────────────────────────────┤
│  ソース C: 自アプリ行動（新規・Phase 2）                 │
│    - watchlist / detail_view イベント                    │
│    - 類似ユーザーおすすめ用（匿名集計）                   │
└─────────────────────────────────────────────────────────┘

対象外: SNS クロール、非スニーカーカテゴリ
```

## 5. 実装フェーズ

### Phase 1 — 既存基盤のフィルタ強化（短期）

- [ ] シルエット/モデルフィルタ（All Star、One Star）
- [ ] コラボキーワード追加（KITH、Brooks Brothers、BEAMS）
- [ ] 設定 UI にシルエット選択を追加
- [ ] ダッシュボード空状態・件数表示の維持

**工数:** 小  
**依存:** なし（KicksDB 商品名にキーワードが含まれる前提）

### Phase 2 — 抽選ページ + 行動ベースおすすめ（中期）

- [ ] `radar_lottery_sources`（抽選 URL 登録・管理者 or 設定画面）
- [ ] Edge Function 定期ジョブ：抽選ページ取得・パース
- [ ] カタログと抽選情報の紐付け（SKU / モデル名）
- [ ] `user_events` または `item_view_logs` テーブル
- [ ] `show_similar_users` 設定 + 簡易レコメンド表示

**工数:** 中〜大  
**リスク:** 抽選ページはサイトごとに HTML 構造が異なる → ソース別パーサが必要

### Phase 3 — Google カレンダー（確認付き）（中期）

- [ ] Google OAuth 連携（Calendar スコープ）
- [ ] トークン安全保管（Supabase Vault / encrypted column）
- [ ] 詳細 or 通知から「カレンダーに追加？」モーダル
- [ ] イベント作成 API（発売日・タイトル・リンク）

**工数:** 中  
**前提:** Google Cloud プロジェクト、OAuth 同意画面

## 7. 抽選ページ監視 — 設計メモ

### 対象例（要個別調査）

| ストア/ブランド | 抽選ページ例   | 取得方式候補                                |
| --------------- | -------------- | ------------------------------------------- |
| Nike            | SNKRS          | 非公式 API 禁止 → 公式情報/RSS/手動登録 URL |
| Converse        | 国内抽選ページ | HTML パース or 手動フィード                 |
| BEAMS / KITH    | コラボ抽選告知 | サイト別パーサ                              |

### 原則

1. **利用規約を確認**してからクロール方針を決める
2. パーサが壊れたときの **アラート・手動フォールバック**
3. KicksDB カタログと **重複排除**（`external_id` + `source`）

---

## 8. 類似ユーザーおすすめ — 設計メモ

### 収集する行動（最小）

- ウォッチリスト追加
- 詳細ページ閲覧（同一 user_id + sneaker_id + timestamp）

### 表示ロジック（初期）

```
IF show_similar_users = true:
  ダッシュボードに「他のユーザーも注目」セクション
  - 自分の brands/silhouettes と重なる商品のうち
  - 直近 7 日で watchlist 追加数が多い Top N
```

### プライバシー

- 個人特定不可の集計のみ
- ログイン必須（authenticated のみ記録）

---

## 9. Google カレンダー — 確認フロー

```
発売通知 or 詳細画面
  → 「Google カレンダーに追加しますか？」
  → [追加する] [後で] [連携しない]
  → 追加する: Calendar API events.insert
     - title: {brand} {model_name}
     - date: release_date or lottery_deadline
     - description: store_url / 抽選 URL
```

完全自動は行わない（ユーザ回答 #5）。

## 10. 成功指標（案）

| 指標                   | 目標                                                   |
| ---------------------- | ------------------------------------------------------ |
| 条件マッチ商品の見逃し | ウォッチ + 通知で発売日当日までリマインド              |
| タイムライン疲れ       | ダッシュボードは常にフィルタ結果のみ（時系列混在なし） |
| 抽選情報               | 登録ソースから 24h 以内にカタログ反映                  |
